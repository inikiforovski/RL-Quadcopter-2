Project: Train a Quadcopter How to Fly¶

Design an agent to fly a quadcopter, and then train it using a reinforcement learning algorithm of your choice!

Try to apply the techniques you have learnt, but also feel free to come up with innovative ideas and test them.

Instructions¶

Take a look at the files in the directory to better understand the structure of the project.

  • task.py: Define your task (environment) in this file.
  • agents/: Folder containing reinforcement learning agents.
    • policy_search.py: A sample agent has been provided here.
    • agent.py: Develop your agent here.
  • physics_sim.py: This file contains the simulator for the quadcopter. DO NOT MODIFY THIS FILE.

For this project, you will define your own task in task.py. Although we have provided a example task to get you started, you are encouraged to change it. Later in this notebook, you will learn more about how to amend this file.

You will also design a reinforcement learning agent in agent.py to complete your chosen task.

You are welcome to create any additional files to help you to organize your code. For instance, you may find it useful to define a model.py file defining any needed neural network architectures.

Controlling the Quadcopter¶

We provide a sample agent in the code cell below to show you how to use the sim to control the quadcopter. This agent is even simpler than the sample agent that you'll examine (in agents/policy_search.py) later in this notebook!

The agent controls the quadcopter by setting the revolutions per second on each of its four rotors. The provided agent in the Basic_Agent class below always selects a random action for each of the four rotors. These four speeds are returned by the act method as a list of four floating-point numbers.

For this project, the agent that you will implement in agents/agent.py will have a far more intelligent method for selecting actions!

import random

class Basic_Agent():
    def __init__(self, task):
        self.task = task
    
    def act(self):
        new_thrust = random.gauss(450., 25.)
        return [new_thrust + random.gauss(0., 1.) for x in range(4)]

The Task¶

A sample task has been provided for you in task.py. Open this file in a new window now.

The __init__() method is used to initialize several variables that are needed to specify the task.

  • The simulator is initialized as an instance of the PhysicsSim class (from physics_sim.py).
  • Inspired by the methodology in the original DDPG paper, we make use of action repeats. For each timestep of the agent, we step the simulation action_repeats timesteps. If you are not familiar with action repeats, please read the Results section in the DDPG paper.
  • We set the number of elements in the state vector. For the sample task, we only work with the 6-dimensional pose information. To set the size of the state (state_size), we must take action repeats into account.
  • The environment will always have a 4-dimensional action space, with one entry for each rotor (action_size=4). You can set the minimum (action_low) and maximum (action_high) values of each entry here.
  • The sample task in this provided file is for the agent to reach a target position. We specify that target position as a variable.

The reset() method resets the simulator. The agent should call this method every time the episode ends. You can see an example of this in the code cell below.

The step() method is perhaps the most important. It accepts the agent's choice of action rotor_speeds, which is used to prepare the next state to pass on to the agent. Then, the reward is computed from get_reward(). The episode is considered done if the time limit has been exceeded, or the quadcopter has travelled outside of the bounds of the simulation.

In the next section, you will learn how to test the performance of an agent on this task.

The Agent¶

The sample agent given in agents/policy_search.py uses a very simplistic linear policy to directly compute the action vector as a dot product of the state vector and a matrix of weights. Then, it randomly perturbs the parameters by adding some Gaussian noise, to produce a different policy. Based on the average reward obtained in each episode (score), it keeps track of the best set of parameters found so far, how the score is changing, and accordingly tweaks a scaling factor to widen or tighten the noise.

Define the Task, Design the Agent, and Train Your Agent!¶

Amend task.py to specify a task of your choosing. If you're unsure what kind of task to specify, you may like to teach your quadcopter to takeoff, hover in place, land softly, or reach a target pose.

After specifying your task, use the sample agent in agents/policy_search.py as a template to define your own agent in agents/agent.py. You can borrow whatever you need from the sample agent, including ideas on how you might modularize your code (using helper methods like act(), learn(), reset_episode(), etc.).

Note that it is highly unlikely that the first agent and task that you specify will learn well. You will likely have to tweak various hyperparameters and the reward function for your task until you arrive at reasonably good behavior.

As you develop your agent, it's important to keep an eye on how it's performing. Use the code above as inspiration to build in a mechanism to log/save the total rewards obtained in each episode to file. If the episode rewards are gradually increasing, this is an indication that your agent is learning.

Reflections¶

Question 1: Describe the task that you specified in task.py. How did you design the reward function?

Answer: The task I am trying to accomplish is for the quadcopter to reach a given target position. I picked this task specifically because it is one of the harder tasks you can train as the coordinates in the space play a big role, as opposed to take off, landing and hovering.

To make the goal more attainable, I added a termination condition when the copter is within 5m of the target in either direciton. The likelihood of exactly hitting a point in the space with continuous state and action spaces is low so this was deemed as a "good enough" solution.

Something I played around with quite a bit was including velocities/angular velocities as part of the state space. The thinking behind this was that a policy shouldn't treat a quadcopter zooming past a given xyz coordinate at high velocities the same way it does that copter doing a slow hover. However, this does increase the state space to 36 variables instead of 18 with the action repeats and I did not attain meaningful learning behavior with this approach.

One more variable I played around with was the runtime - I set the maximum episode length to 30s instead of the given 5s to see if this would help the copter recover from bad trajectories and reach the goal given a longer timespan. I stuck to this runtime for a while but in the end found that it gave too much leeway for the model to explore the space without making progress towards the goal at the outset. Perhaps this would have worked better if I left the model to train for several thousand episodes but given time constraints and feedback I received from several mentors about how long a training session should be I decided to go back to the shorter total episode length.

It took a very long time to design a reward function that worked for the task of reaching a given position. I tried many different iterations and versions, using tanh, exponential decay, the simple linear function, stepped reward, etc. For the final function, the main component is the distance reward that I have included based on suggestions from https://www.youtube.com/watch?v=0R3PnJEisqk. It takes the distance to target and scales it by the absolute furthest distance the drone can be from that target in each dimension. This allows for stronger penalties the further the drone is instead of a simple linear function.

I have included two further additions - a big reward for getting within the range of the coordinates and a sizable penalty when the copter is going beyond the boundaries of the environment. This was a big part in reinforcing the nature of the task - not crashing/flying too far and achieving the target.

Question 2: Discuss your agent briefly, using the following questions as a guide:

  • What learning algorithm(s) did you try? What worked best for you?
  • What was your final choice of hyperparameters (such as $\alpha$, $\gamma$, $\epsilon$, etc.)?
  • What neural network architecture did you use (if any)? Specify layers, sizes, activation functions, etc.

Answer: As outlined in the project, I tried the DDPG agent presented. I also attempted to use two other DDPG agent implementations from Github: https://pemami4911.github.io/blog/2016/08/21/ddpg-rl.html and https://github.com/floodsung/DDPG. The end result is farily similar and I ended up switching to the udacity provided DDPG agent so I could get a better understanding of how other students were tackling the project in the Student Hub.

I used hyperparameters provided in the original DDPG paper, with the actor and critic networks having two main branches of 400 and 300 units, with a further 300 units for the acion pathway in critic network. Upon trying out several architectures, I found that having a dropout layer and batch normalization in the critic network gave the agent a better chance at exploring the space and diverging to a good policy. Batch normalization in particular was very beneficial, it prevented the model form being stuck in a suboptimal policy.

I spent many hours tweaking the hyperparameters but did not get significantly different results from the base set. In the end I tweaked LR to be 0.0001 for both actor and critic networks, changed gamma to 0.9 and halved the OU theta and sigma to 0.075 and 0.1.|

Question 3: Using the episode rewards plot, discuss how the agent learned over time.

  • Was it an easy task to learn or hard?
  • Was there a gradual learning curve, or an aha moment?
  • How good was the final performance of the agent? (e.g. mean rewards over the last 10 episodes)

Answer: Once the correct architecture was in place, the task became more or less easy to learn. Before that moment it was next to impossible. There are certain runs I do that still fail to converge for a long time if they do not hit the "aha" moment where they land on a policy that takes them close to the goal. I did not see a slow gradual learning curve. This is also in part due to the nature of how I structured the rewards - there is a very high bonus for hitting the target so the agent is likely to stick to a policy that got it there once it reaches the goal a couple of times.

In terms of evaluating performance, I relied more on the agent achieving its goal and terminating as well as 3D visual plots of what it was doing instead of looking at mean rewards. I did this for a couple of reasons - the total mean reward for an episode varies greatly depending on how long the episode is and is therefore not very telling - a short episode where the copter just falls down can have a better looking reward than a longer episode where it flies but gets further away from the target. Additionally, due to the "aha" moment nature of learning, the rewards plot tends to be pretty simple and does not explain well what is going on within the model.

Reward per episode is a better metric but again not 100% reliable - if it takes longer for the copter to get to its reward the total average reward is smaller but the task was still completed.

Question 4: Briefly summarize your experience working on this project. You can use the following prompts for ideas.

  • What was the hardest part of the project? (e.g. getting started, plotting, specifying the task, etc.)
  • Did you find anything interesting in how the quadcopter or your agent behaved?

Answer: This was all in all an very time consuming project. There was little guidance to start off with and very little support from the community, especially with slack deprecated and the Student Hub in its place.

It was also very tedious in the sense that there was very little I was tweaking in terms of the code and the hyperparameters. The results were random and hard to interpret, especially in trying to discern whether or not I was getting closer to a good model. Many hours were spent waiting for enough episodes to elapse so I could once again see that the agent was getting nowhere.

There was no real instruction in terms of how to measure performance of the model beside the episode plot and the positional plot. I spent a good amount of time coming up with how to properly measure what was going on so I could get feedback as the agent was training.

In the end, I am saving several files for analysis - a CSV with the testing episode data, png files showing the 3D trajetory of the quadcopter and plots for the total and the average rewards per episode over training episodes.

Something that seemed very strange to me is no mattter how much I played around with the architecture, reward function and hyperparameters, the rotor speeds would always converge at 0 or 900, especially towards the end of a given task. In the end, even though most of the tasks I tested were a vertical rise, the policy would often converge on only two rotors spinning at the same time (I clipped rotor speeds to [1,900] because speeds close to 0 and over 900 had weird effects on physics_sim). It would be intersting to find out what kind of rotor speeds other students and mentors witnessed and why this "desire" to minimize or maximize speed was so inherent in this task.

One good idea I found in another DDPG implementation was recording performance based on test episodes instead of training episodes. This consists of training the model for 10 episodes and then running 10 instances of the model that have the same policy. This shows whether or not the policy results in stable trajectories or not and I found that to be a good extra layer of validation.

Determining a proper reward function and comparing different reward functions against each other was another really time consuming process. I would say the most time was spent tweaking the reward function and trying to understand what effect it was having on the model. There was a whole array of reward functions previous students had posted which was interesting to explore but ultimately only the one I am using gave me decent results for the task I chose.

To end on a positive note, this was a good learning exercise in model visualization and tracking. I acquired several skills necessary to more accurately eveluate reinforcement learning models beyond just looking at the reward function plot.
